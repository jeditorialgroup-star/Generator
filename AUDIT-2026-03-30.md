## 5. Bug naturalizer — RESUELTO

### Causa raíz

El pipeline blog-generator + naturalizer generaba duplicados por dos bugs:

#### Bug #1 — naturalizer.py: falta `'future'` en queries SQL
**Archivo:** `/home/devops/projects/inforeparto/naturalizer/naturalizer.py`

- Línea 233: query de `process_post()` solo buscaba `IN ('publish','draft')` → posts `future` no se naturalizaban
- Línea 297: query de `get_modified_today_ids()` mismo problema

**Fix aplicado:** Añadido `'future'` a ambas queries.

#### Bug #2 — publisher.py: siempre crea nuevo post sin verificar slug
**Archivo:** `/home/devops/projects/inforeparto/blog-generator/blog-generator/src/publisher.py`

- El método `publish_prepared()` siempre hacía `POST /posts` (crear nuevo), sin comprobar si ya existía un post con el mismo slug
- Si el generador se ejecutaba dos veces, se creaban dos posts idénticos

**Fix aplicado:** Antes de crear, se busca por slug vía `GET /posts?slug=X&status=any`. Si existe, hace `POST /posts/{id}` (actualizar) en lugar de crear nuevo.

#### Bug #3 — Sin sincronización entre script manual y blog-generator
El script `amazon-flex-posts.sh` crea posts manualmente con WP-CLI mientras el blog-generator también puede procesarlos. No hay mecanismo que los coordine.

**Recomendación:** Usar el blog-generator como único pipeline de publicación. Si se usa el script manual, eliminar el tema del queue del blog-generator antes de publicar.
